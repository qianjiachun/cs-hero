# Git Commit

## Core Rule

Only create a commit when the user explicitly asks for a commit.

If the user asks to "save", "finish", "update files", "implement", or "fix" without mentioning commit, do not commit. Summarize the changed files and ask before committing.

## Author & Contributor Policy (Hard Rule)

This repository has **one human maintainer only**. The agent assists with code; it is **not** a project contributor.

**Never** do any of the following unless the user gives an explicit, one-off override for that specific commit:

- Add commit trailers such as `Co-authored-by`, `Signed-off-by`, `Assisted-by`, or similar lines naming AI, Cursor, Composer, Copilot, agents, bots, or `ai-agent`.
- Use `git commit --author`, or set `GIT_AUTHOR_NAME`, `GIT_AUTHOR_EMAIL`, `GIT_COMMITTER_NAME`, or `GIT_COMMITTER_EMAIL` to an AI or synthetic identity.
- Add or edit `CONTRIBUTORS`, `AUTHORS`, `CODEOWNERS`, `package.json` `author` / `contributors`, or docs that list an AI or `ai-agent` as a project contributor.
- Write commit message bodies that credit AI as a co-author or maintainer (for example "Co-authored-by: Cursor", "Generated by AI").

### No AI Names in Commit Messages (Hard Rule)

The **subject and body** must not contain AI product or vendor names, including but not limited to:

`AI`, `Cursor`, `Composer`, `Copilot`, `ChatGPT`, `GPT`, `Claude`, `Gemini`, `OpenAI`, `Anthropic`, `ai-agent`, `agent`, `bot`, `LLM`, or spelling variants.

Describe the change in neutral technical terms instead.

Bad:

```text
chore: add git-commit cursor skill for AI commits
docs: fix Copilot-generated README
```

Good:

```text
chore: add git-commit skill for commit conventions
docs: fix README install steps
```

Before `git commit`, scan the full message (subject + body) and rewrite if any forbidden name appears.

**Always**:

- Use the repository's normal git identity from local config (`user.name` / `user.email`). Do not change git config.
- Keep commit messages free of AI attribution trailers and AI product names.
- If the user asks to "add AI as contributor" or similar, decline and explain that project policy allows only the human maintainer; offer to commit under the user's identity only.

Before `git commit`, read the full message and reject it if any forbidden attribution appears.

**What this skill cannot do**: It binds the AI agent only. It does not stop the human from typing trailers manually, other tools, or GitHub UI. For machine enforcement, add a `commit-msg` hook separately (only if the user requests it).

## Before Committing

Run these checks before staging anything:

```bash
git status --short
git diff -- .
git diff --cached -- .
git log --oneline -5
```

Use the results to identify:

- Which changes were made by the agent in this task.
- Which changes appear unrelated or user-authored.
- Whether any files may contain secrets, credentials, local data, binaries, generated artifacts, logs, or build outputs.
- The repository's recent commit message style.

Do not stage unrelated changes. Do not revert unrelated changes.

## Staging Rules

Stage only files that are relevant to the requested commit.

Never stage these unless the user explicitly confirms:

- `.env`, `.env.*`, credentials, tokens, private keys, cookies, local config with secrets.
- Runtime data such as `dev/`, logs, caches, recordings, generated videos, `release/`, `dist/`, `out/`, `node_modules/`, `vendor/`.
- Large binaries, downloaded runtime archives, or generated media.
- Plan files under `.cursor/plans/` unless the user specifically asked to change plans.

If a file contains both relevant and unrelated changes, ask the user before committing, or stage only the relevant hunks if safe and non-interactive tooling is available.

## Validation

Run the narrowest useful verification before committing:

- For TypeScript changes: `pnpm exec vue-tsc --noEmit`
- For build/tooling changes: `pnpm exec electron-vite build`
- For dependency or runtime install script changes: `pnpm install` only when needed and reasonable.

If verification cannot be run, say so in the final response and explain why.

Do not skip hooks with `--no-verify` unless the user explicitly requests it.

## Commit Message

Prefer concise English commit messages in the existing repository style.

Use:

```text
<type>: <imperative summary>
```

Common types:

- `feat`: new user-visible capability.
- `fix`: bug fix.
- `chore`: dependencies, scripts, build plumbing, repository maintenance.
- `docs`: documentation only.
- `refactor`: behavior-preserving code restructuring.
- `test`: tests only.

The summary should describe the intent, not a file list. Never put AI product names in the summary or body (see **No AI Names in Commit Messages** above).

Examples:

```text
fix: stabilize CS2 game capture selection
chore: automate runtime dependency setup
docs: document contributor install flow
```

For multi-line messages, keep the body short and explain why the change exists.

Always pass commit messages via a heredoc:

```bash
git commit -m "$(cat <<'EOF'
chore: automate runtime dependency setup

Download required runtime binaries during install so contributors can run the app after cloning.
EOF
)"
```

## Safety Rules

Never run destructive git commands unless the user explicitly asks and confirms the risk:

- `git reset --hard`
- `git clean -fd`
- `git checkout -- <path>`
- `git restore <path>`
- `git rebase`
- `git push --force`

Never push unless the user explicitly asks.

Never amend unless all are true:

- The user requested amend, or the last commit was created by this agent in the current conversation and a hook modified files that must be included.
- The commit has not been pushed.
- The amended changes are only part of the same requested commit.

If a commit fails because of a hook, fix the issue and create a new commit. Do not amend a failed or rejected commit.

## Final Response After Commit

Report:

- Commit hash and message.
- Verification commands run and their results.
- Any files intentionally left unstaged.
- Any risks or follow-up steps.

Keep the response short and do not paste full diffs unless the user asks.
